# Product-Eval Gold Set

Calibration anchor for Rouge's product-quality judge
(`src/prompts/loop/02e-evaluation.md` PO lens) against the rubric at
`library/rubrics/product-quality-v1.md`.

## What this exists for

A judge prompt can silently drift. Tests pass, JSON shapes remain valid,
but the distribution of verdicts moves. This directory is the fixed point
that future prompt changes are measured against. `rouge eval-calibrate`
runs the judge against these fixtures and requires quadratic-weighted
Cohen's Kappa ≥ 0.75 per rubric dimension before a prompt change to the
judgment layer is considered safe to promote.

## Ground truth: synthetic, not human-labeled

The initial 20 entries (`synthetic-01.json` … `synthetic-20.json`) are
generated by `src/launcher/gold-set-synth.js`. Each fixture has a
`cycle_context_excerpt` whose fields are rule-derived from target rubric
scores — e.g. a fixture with `journey_completeness: 3` has
`product_walk.journeys[]` where every journey reaches a visible success
state; a fixture with `interaction_fidelity: 0` has every interactive
element return `"no visible response"`.

Labels for synthetic fixtures carry `labeler: "synthetic-v1"`.

**Why synthetic instead of human labels.** The owner is not a
data-labeling worker. A gate that requires hand-labeled bootstrap data
is permanently dormant — that design was tried and corrected. Synthetic
ground truth is frozen-in-time: if rubric v1 anchors change meaning,
regenerate fixtures against the new anchors (rubric_version bump → new
generator → new fixtures).

What synthetic calibration catches: a prompt edit that makes the judge
score the same cycle_context_excerpt differently from before. That is the
drift class that matters — silent scoring shifts under small prompt
tweaks. What it doesn't catch: drift where the generator rules themselves
were wrong. The generator is open code; anyone reviewing it can challenge
the anchor-to-signal mapping and regenerate.

Non-synthetic entries (human-labeled, cross-model reference, whatever)
are permitted and will not be clobbered by the generator — `rouge
eval-seed-gold` only overwrites entries whose labeler is `synthetic-v1`.

## Regenerate the synthetic set

```bash
rouge eval-seed-gold           # overwrite synthetic entries, preserve others
rouge eval-seed-gold --dry-run # show what would change
rouge eval-seed-gold --force   # also clobber non-synthetic entries (careful)
```

The generator is deterministic: same version → byte-identical output.
Running it twice produces an empty git diff.

## Who edits this directory

**Humans authoring non-synthetic entries, or Rouge's own generator.**
The self-improve pipeline's blocklist
(`rouge.config.json` → `self_improvement.blocklist` → `library/gold-sets/**`)
prevents the LLM pipeline from authoring or silently editing entries. If
the pipeline could edit them, it could quietly retune labels to match
drifting model output, and every calibration would pass by construction.

## Entry shape

See `schemas/gold-set-entry-v1.json`. Each entry:

```json
{
  "schema_version": "gold-set-entry-v1",
  "id": "synthetic-01",
  "rubric_id": "product-quality",
  "rubric_version": 1,
  "source": {
    "project": "synthetic",
    "cycle_id": "synthetic-01",
    "captured_at": "2026-04-23T00:00:00Z"
  },
  "cycle_context_excerpt": {
    "product_walk": { "journeys": [...], "screens": [...] },
    "design_review": { "category_scores": {...}, "copy_quality": "...", "ai_slop_score": 0.1 },
    "functional_correctness": "PASS",
    "console_errors": 0,
    "active_spec_fit": "observed product delivers the stated value proposition..."
  },
  "human_labels": {
    "journey_completeness": 3,
    "interaction_fidelity": 3,
    "visual_coherence": 3,
    "content_grounding": 3,
    "edge_resilience": 3,
    "vision_fit": 3
  },
  "human_verdict": "PRODUCTION_READY",
  "labeler": "synthetic-v1",
  "labeled_at": "2026-04-23T00:00:00Z",
  "notes": "..."
}
```

Scores are integers `0..3` per rubric anchor, or `null` to abstain.
Abstain excludes that dimension from the calibration for this entry.
`human_verdict` is derived by the rubric's aggregation rules (any 0 →
`NOT_READY`; any 1 with no 0 → `NEEDS_IMPROVEMENT`; every ≥ 2 with at
least one 3 → `PRODUCTION_READY`; else `NEEDS_IMPROVEMENT`).

## How the calibrator uses these

```bash
rouge eval-calibrate --model-labels <path>
```

- Loads every `*.json` entry here.
- `--model-labels` is a JSON object keyed by entry id, each value
  `{labels: {<dim>: score}, verdict: "..."}` — produced by running the
  judge against each fixture's `cycle_context_excerpt` and extracting
  `evaluation_report.po.rubric_scores`.
- Computes quadratic-weighted Cohen's Kappa per dimension.
- Exits 0 if every dimension ≥ `eval_calibration.min_kappa` AND
  `eval_calibration.min_entries` paired entries exist.
- Exits 1 if any dimension fails or collapses.
- Exits 2 if there's not enough data yet.

A score-distribution invariant holds for the default synthetic set:
every dimension takes at least three distinct values across the 20
entries, so no dimension can collapse. The verdict distribution spans
all three verdicts. Both properties are asserted by tests.
